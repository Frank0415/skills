# cspec-zh

用受控技术中文编写、改写或审阅软件规范。适用于需求说明、API 契约、设计规格、运行手册、操作步骤、配置参考、验收条件及其他不能容忍歧义的技术文档。

本技能借鉴 ASD-STE100 的受控语言思路、BCP 14 的规范性关键词和中文标准文件的严谨表达方式，并针对软件工程补充代码标识符、接口、状态、错误和测试规则。它不是 ASD-STE100 的官方中文版本，也不代表任何国家标准认证。

## 目标

让开发者、审阅者、测试人员和运维人员对同一句规范得到相同解释。每项要求应能明确回答：

- 谁或哪个组件执行动作；
- 在什么条件下执行；
- 必须、应该或可以做什么；
- 输入和初始状态是什么；
- 可观察结果是什么；
- 如何验证结果。

简化语言，不删减技术内容。

## 写作前

先检查代码、模式定义、接口、现有术语和源文档。项目已经使用的准确术语优先于本技能的偏好。

确定会影响规范的内容：

- 读者和文档类型；
- 系统边界、组件和参与者；
- 权威术语、代码标识符和版本；
- 单位、格式、范围、默认值和空值语义；
- 规范性要求与解释性内容；
- 兼容性、安全、权限和数据丢失约束；
- 验收或验证方法。

能从材料中查明的内容不要询问。缺失信息会改变契约时，提出最小问题。暂时无法决定的值用 `TBD` 标记，并注明负责人或待决问题。不要为了让文档显得完整而编造参数。

## 规范性用语

软件规范默认使用以下关键词。文档首次使用时应声明其含义：

- `必须`：符合规范的实现必须满足该要求。
- `不得`：符合规范的实现绝对禁止该行为。
- `应该`：强烈推荐；存在合理例外时可以偏离。
- `不应该`：通常禁止；存在合理例外时可以偏离。
- `可以`：允许实现选择该行为，不构成要求。
- `可选`：功能、字段或步骤不是必需项。

`应该` 和 `不应该` 若存在重要例外，应写出例外或判断条件。所有合规实现都必须满足的行为使用 `必须`，不要使用 `应该` 降低强度。偏好和审美选择不要写成 `必须`。

不要把“必须、应当、应、需要、需、应该、建议、最好、尽量”混作同一级要求。选定一套关键词后全文一致。项目若要求 GB/T、法规、合同或企业词表，以项目规定的“应、宜、可”等词义为准，不要同时混用本技能的默认词表。

双语规范可以在首次出现时并列 `MUST（必须）`、`SHOULD（应该）` 和 `MAY（可以）`。之后选定一种语言，不要来回切换要求等级。

## 受控中文

### 术语

- 一个概念只用一个名称，一个名称只表达一个概念。
- 沿用项目中的正式术语，不为避免重复而替换同义词。
- 新术语首次出现时定义，或加入术语表。
- 产品名、API 名、类型名、字段名、枚举值、命令、路径、字面量和单位保持原样。
- 使用直接动词。写“验证令牌”，不要写“对令牌进行验证操作”。
- 使用具体名词。写“认证服务”，不要写“相关模块”或“该部分”。
- 缩写可能不明确时，首次出现给出全称。项目已将缩写作为正式名称时无需机械展开。

### 句子

- 一个说明句只讲一个主题。
- 一个规范句只包含一项可独立测试的要求。
- 一个操作句只包含一个动作；同时发生且不可拆分的动作可以放在一句。
- 明确写出执行者。主语可能混淆时，不要依赖中文省略主语的习惯。
- 操作步骤以动词开头。
- 条件先于行为：`如果令牌已过期，认证服务必须返回……`。
- 时序用“在……之前”“在……之后”“仅当……时”明确表达。
- 长逻辑拆成短句或纵向列表。不要为了缩短句子而删掉主语、动词、对象或条件。
- 不把英文的 20 个单词机械换算成 20 个汉字。中文以“一句一个主题、要求或动作”为准绳。

### 消除歧义

- “它、这、其、该项、上述内容”存在多个先行对象时，重复写出具体名词。
- 删除或量化“相关、相应、适当、及时、尽快、通常、一般、原则上、必要时、等、等等”。
- 不单独使用“正常”“异常”“有效”“合理”“足够”“较大”“较快”。给出状态、阈值或判断规则。
- “和”与“或”可能产生不同结果时，明确写“同时满足”“至少满足一项”或“只能选择一项”。
- 明确范围是否包含端点，时间窗口使用哪个时区，大小使用哪个单位。
- 每个例外紧跟它所限制的要求，不把例外放到远处的备注中。
- 正面写出期望行为。只有真实禁令、非法状态和安全约束使用否定要求。
- 不把要求藏在备注、示例、背景或理由中。

## 软件领域用语

### 避免宽泛动词

以下词在软件文档中容易隐藏真实行为：

- 不写“系统支持 JSON”，写“API 接受 `application/json` 请求体”。
- 不写“服务处理错误”，写“服务记录错误并返回 HTTP `503 Service Unavailable` 状态码”。
- 不写“应用管理会话”，分别写创建、读取、续期、撤销或删除会话。
- 不写“优化性能”，给出延迟、吞吐量、内存或复杂度目标。
- 不写“保证安全”，给出认证、授权、加密、审计或隔离要求。

为同类动作保持固定动词：

- 客户端`发送`请求或消息；服务`返回`值、响应或状态码。
- 程序`读取`和`写入`存储；日志系统`记录`事件。
- 语言运行时`抛出`异常；API `返回`错误，不混用两者。
- 组件`拒绝`非法输入；只有明确无副作用时才写`忽略`。
- `重试`表示再次执行失败操作；必须同时定义次数、间隔和停止条件。

### 代码与标识符

- Markdown 使用反引号，HTML 使用 `code` 元素标记代码实体。
- 类型、类、方法、字段、枚举值、环境变量、文件名、路径、命令和字面量保持精确大小写。
- 不给标识符添加中文词缀。写“`User` 对象”“`close` 方法”“`READY` 枚举值”。
- 写“发送 `POST` 请求”“返回 HTTP `401 Unauthorized` 状态码”，不要把 `POST` 或 `GET` 当中文动词。
- 区分文件名、文件类型、命令、命令输出和 UI 文本。
- UI 标签按界面原文书写；操作动词与控件名称分开，例如“选择 **保存**”。

### 数据语义

不要把以下状态合并为“空”：

- 字段缺失；
- `null`；
- 空字符串 `""`；
- 空数组或空对象；
- 值为 `0` 或 `false`。

对输入和输出按需定义类型、编码、格式、单位、范围、精度、默认值、必填性、可空性、排序和重复项规则。

### 行为契约

对每个适用的接口或行为说明：

- 前置条件和权限；
- 触发条件和输入；
- 状态变化和执行顺序；
- 返回值、输出或外部副作用；
- 失败条件和精确错误行为；
- 超时、重试、取消和幂等性；
- 并发、一致性和事务边界；
- 兼容性和迁移行为；
- 验证方法。

只写适用内容，不为追求模板完整而添加空章节。

### 错误写法

错误要求应同时写出条件和可观察结果。根据接口类型明确状态码、错误码、异常类型、退出码、日志级别或重试行为。

弱：

> 登录失败时应该提示一下并做相应处理。

受控写法：

> 客户端向 `POST /sessions` 发送无效密码时，服务必须返回 HTTP `401 Unauthorized` 状态码。响应体必须包含 `code: "INVALID_CREDENTIALS"`。服务不得创建会话。

## 操作步骤

- 顺序重要时使用编号列表。
- 每一步以明确动词开头。
- 操作位置不明显时，先写位置。
- 独立动作拆成独立步骤。
- 命令和代码必须可以直接识别或执行。
- 后续步骤依赖结果时，写出可观察结果。
- `备注` 只提供信息，不承载操作要求。
- 危险动作之前写警告。警告应说明危险、后果和预防动作。

## 验收条件

每项验收条件都应可重复验证。写明输入、初始状态、动作和结果。Given/When/Then 能改善契约时可以使用，但不要强制所有要求采用同一格式。

示例默认不具规范性，除非文档明确声明。示例必须符合规范条文，不能悄悄增加另一种行为。

## 文档结构

优先保留用户要求的结构和仓库既有模板。新建大型规格时，只选择有助于实现或验证的章节：

```markdown
# 标题

## 目的与范围
## 术语与规范性用语
## 参与者与系统边界
## 功能要求
## 接口与数据
## 状态与失败行为
## 验收条件
## 待决问题
```

需要追踪时使用稳定的需求 ID。一个 ID 只对应一项要求。不要给包含多项独立要求的整段文字分配一个 ID。

规范条文和理由分开。理由可以解释设计决策，但不能削弱或扩大要求。

## 改写流程

1. 保留源文档的事实、约束、技术细节和指定结构。
2. 建立小型术语表，找出同义词漂移。
3. 区分要求、说明、步骤、示例和理由。
4. 为每项要求补齐执行者、条件、直接动词和可观察结果。
5. 统一规范性关键词和代码格式。
6. 用实现、接口或测试证据核对要求；尚未决定的内容标记为提案或 `TBD`。
7. 删除重复、模糊限定、空泛结论和装饰性文字。
8. 确认验收条件覆盖所有规范行为。

审阅任务应先列出歧义、矛盾、不可验证要求、缺失状态和术语冲突，并标明具体章节或行号。用户只要求审阅时，不要擅自改写文件。

## 交付检查

交付前确认：

1. 每个规范句只有一个要求等级和一项可测试行为。
2. 执行者、术语、标识符、单位、条件和范围没有歧义。
3. 操作步骤保持真实顺序并使用直接动词。
4. 需要定义的错误、状态和副作用已经写明。
5. 示例与规范条文一致。
6. 简化语言时没有删除源要求或技术细节。
7. 文档没有把本技能称为官方中文 STE，也没有虚构标准合规性。
